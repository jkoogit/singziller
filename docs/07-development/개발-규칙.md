# Development Guidelines

## 목적

Singziller는 한국어 도메인이 강한 서비스이므로, 개발 코드도 가능한 범위에서 도메인 의미가 바로 읽히도록 작성한다. 단, 외부 API, 표준 라이브러리, 브라우저 API, Google/YouTube/Sheets 필드명처럼 상호운용성이 필요한 이름은 원래 이름을 유지한다.

## 이름 규칙

### 한글 이름 우선

내부 코드의 변수, 함수, 클래스, 모듈 이름은 의미를 살린 한글 이름을 우선한다.

```js
const 검색조건 = 가져오기검색조건();
const 매핑후보 = 노래번호매핑후보만들기(곡목록);

function 곡검색하기(곡목록, 검색조건) {
  return 곡목록.filter((곡) => 조건에맞는곡인가(곡, 검색조건));
}
```

### 영어 이름을 유지하는 경우

아래 항목은 영어를 유지한다.

- DOM 표준: `document`, `querySelector`, `addEventListener`
- Web API: `fetch`, `Response`, `localStorage`, `indexedDB`
- Google/YouTube API 필드: `access_token`, `spreadsheetId`, `valueInputOption`
- DB 컬럼과 JSON API 계약 중 외부 시스템과 직접 맞물리는 이름
- npm package, import/export 이름

### 혼합 이름 허용

외부 API 계약과 내부 의미가 함께 필요할 때는 내부 래퍼를 한글로 만든다.

```js
async function 구글시트값가져오기(시트ID, 범위) {
  return fetch(`https://sheets.googleapis.com/v4/spreadsheets/${시트ID}/values/${범위}`);
}
```

## 코드 작성 원칙

- 코드만 읽어도 동작 의도가 보이게 작성한다.
- 함수는 한 가지 일을 하게 작게 유지한다.
- 조건문은 도메인 의미가 있는 함수로 추출한다.
- 원천 데이터, 정규화 데이터, 사용자 데이터는 이름에서 구분한다.
- 외부 API 응답을 앱 내부에서 바로 쓰지 않고 변환 함수를 거친다.

## 주석 정책

주석은 코드가 말하지 못하는 구조와 의사결정을 설명할 때만 사용한다.

좋은 주석:

```js
// 원천 데이터는 자동 수집 결과이므로 사용자가 확정한 수동 보정을 덮어쓰지 않는다.
if (기존곡.manualLocked) return 기존곡;
```

피할 주석:

```js
// 곡을 반환한다.
return 곡;
```

## 테스트 주도 개발

새 기능, 버그 수정, 구조 변경은 테스트를 먼저 작성한다.

작업 순서:

1. 실패하는 테스트를 먼저 작성한다.
2. 실패 이유가 기대한 기능 부재인지 확인한다.
3. 최소 구현으로 테스트를 통과시킨다.
4. 리팩터링 후 테스트를 다시 실행한다.
5. 작업 리뷰 문서에 검증 결과를 기록한다.

테스트 대상:

- 정규화 규칙
- 곡 매핑 점수 계산
- 카탈로그 delta 적용
- IndexedDB 저장/조회 래퍼
- API 요청 URL/페이로드 생성
- 수집 raw record 중복 판정

## 작업 리뷰 문서

모든 의미 있는 작업은 `docs/reviews/` 아래에 리뷰 문서를 만든다.

파일명:

```text
YYYY-MM-DD-작업명-review.md
```

포함 내용:

- 작업 목적
- 변경 파일
- 의사결정
- 테스트/검증 결과
- 남은 위험
- 다음 작업

## 기존 코드 정리 방침

기존 영어 이름 코드를 한 번에 전부 바꾸지 않는다. 기능 변경이나 리팩터링을 하는 파일부터 점진적으로 한글 이름을 적용한다. 외부 API 계약과 테스트가 깨질 가능성이 있는 변경은 반드시 테스트를 먼저 보강한 뒤 진행한다.
