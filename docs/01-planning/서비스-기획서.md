# 서비스 기획서

## 서비스명

노래질러 Singziller

## 서비스 목적

TJ와 KY 노래방 곡정보를 기준으로 사용자가 노래를 검색하고, 브랜드별 번호 차이와 버전 차이를 확인하며, 레파토리와 북마크, 녹음 자료를 오프라인에서도 관리할 수 있는 PWA 서비스를 제공한다.

## 문제 정의

- TJ와 KY는 같은 곡이라도 노래번호가 다르다.
- 금영은 MR, 반주, 라이브 등 여러 버전이 등록될 수 있다.
- 공식 검색은 온라인에 의존하고, 개인 레파토리 관리는 따로 분산되어 있다.
- 사용자는 노래방 현장에서 빠르게 번호를 찾고, 자주 부르는 곡을 오프라인에서도 확인하고 싶다.
- 곡정보는 시간이 지나며 최신곡, 삭제곡, 번호 변경, 버전 추가가 발생한다.

## 서비스 방향

Singziller는 PWA 앱과 독자 곡정보 플랫폼으로 나눈다.

- PWA 앱: 검색, 레파토리, 북마크, 녹음, 오프라인 사용.
- PostgreSQL 카탈로그 서버: 정규화된 독자 곡정보 제공.
- 수집 모듈: TJ/KY 공홈, YouTube, 개인 수집 API, Google Sheet, 수동 입력을 정기적으로 수집.
- 로컬 카탈로그: 필요한 scope와 version만 IndexedDB에 내려받아 저장.

## 핵심 사용자

- 노래방을 자주 이용하는 개인 사용자.
- TJ/KY 번호를 비교해야 하는 사용자.
- 공연/모임/동호회 레파토리를 관리하는 사용자.
- 직접 곡정보를 수집하거나 검수하는 운영자.

## 핵심 가치

- 브랜드별 번호 비교.
- 오프라인 사용성.
- 개인 레파토리 관리.
- 신뢰 가능한 독자 곡정보 카탈로그.
- 버전별 카탈로그 다운로드.
- 수집/검수 기반 현행화.

## MVP 범위

- 정적 PWA 앱.
- 로컬 검색과 샘플 곡정보.
- 레파토리/북마크/녹음.
- Google Sheets import/export.
- 카탈로그 정책 문서화.
- PostgreSQL 기반 카탈로그 서버 설계.

## 확장 범위

- PostgreSQL 서버와 카탈로그 API.
- IndexedDB 기반 로컬 카탈로그 다운로드.
- TJ/KY/YouTube/개인 API 수집기.
- 관리자 검수 큐.
- 검색 품질 고도화.
- 카탈로그 delta 업데이트.

## 성공 지표

- 오프라인에서 다운로드된 scope 검색 가능.
- TJ/KY 번호 매핑 후보 표시율.
- 검색 결과 응답 속도.
- 카탈로그 업데이트 성공률.
- 수집 실패율과 검수 대기 건수.
- 사용자가 저장한 레파토리/북마크 유지율.

## 의사결정 필요

- 서비스 서버를 어떤 스택으로 만들지 결정해야 한다.
- 공식 크롤링 범위와 정책을 확정해야 한다.
- 카탈로그 최초 배포 포맷을 결정해야 한다.
- 관리자 검수 UI를 MVP에 포함할지 결정해야 한다.

## 보완 필요

- TJ/KY 공식 검색 접근 방식 상세 조사.
- 실제 곡 데이터 크기 추정.
- 사용자별 계정/동기화 정책 구체화.
- 법적/약관 리스크 확인.
