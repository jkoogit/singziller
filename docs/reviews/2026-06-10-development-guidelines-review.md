# 개발 규칙 정리 리뷰

## 작업 목적

Singziller 개발 시 한글 변수/함수명 사용, 구조 설명 주석, 테스트 주도 검증, 작업별 리뷰 문서 작성 원칙을 프로젝트 문서로 고정한다.

## 변경 파일

- `docs/07-development/개발-규칙.md`
- `docs/reviews/review-template.md`
- `docs/reviews/2026-06-10-development-guidelines-review.md`
- `README.md`

## 의사결정

- 내부 도메인 코드는 한글 이름을 우선한다.
- 외부 API, 표준 Web API, DB/API 계약 필드는 상호운용성을 위해 영어를 허용한다.
- 기존 코드를 한 번에 전면 리네이밍하지 않고, 기능 변경이 발생하는 파일부터 점진적으로 한글 이름을 적용한다.
- 테스트를 먼저 작성하고 실패를 확인한 뒤 구현하는 흐름을 기본으로 한다.
- 의미 있는 작업마다 `docs/reviews/` 아래에 리뷰 문서를 남긴다.

## 테스트 / 검증

- 문서 추가 작업이라 실행 테스트는 필요하지 않다.
- 이후 코드 변경 작업에서는 해당 기능 테스트를 먼저 작성해야 한다.

## 남은 위험

- JavaScript 한글 식별자는 문법적으로 가능하지만, 외부 도구 일부에서 표시/검색/정렬 이슈가 생길 수 있다.
- API 계약 필드까지 한글화하면 연동 오류가 발생할 수 있으므로 경계가 필요하다.

## 다음 작업

- 다음 코드 변경부터 한글 이름 정책을 적용한다.
- 기존 `app.js` 리팩터링을 진행할 경우 테스트 보강 후 단계적으로 함수명을 한글화한다.
