# 에이전트 작업 운영

## 목적

Singziller 작업을 Codex 에이전트 세션에서 반복 가능하게 진행하기 위한 운영 기준이다. 작업은 기획, 설계, 구현, 리뷰, 테스트, 평가, 일정관리 흐름으로 나누고, 결과는 GitHub Issue, PR, 문서, 리뷰 기록에 남긴다.

## 기본 단위

```text
GitHub Issue  -> 실행 단위
docs          -> 기획, 설계, 운영 기준
작업 브랜치   -> 구현 단위
PR            -> 검토와 dev 반영 단위
docs/reviews  -> 작업 평가와 검증 기록
```

작업 브랜치는 `dev`를 기준으로 만들고, PR 대상도 `dev`로 둔다.

## 역할 흐름

### 1. 기획

- 작업 목적과 사용자 가치를 정리한다.
- 범위를 한 세션에서 끝낼 수 있는 크기로 줄인다.
- 결정이 필요한 항목과 보완 조사를 분리한다.
- 결과는 GitHub Issue 또는 `docs/08-project-management/다음-작업-체크리스트.md`에 남긴다.

### 2. 설계

- 변경할 모듈, 파일, 데이터 흐름, 실패 처리 방식을 정한다.
- API, DB, 외부 연동, 보안값 사용 여부를 명시한다.
- 불확실한 외부 의존성은 stub 또는 mock으로 격리한다.
- 중요한 설계는 `docs/superpowers/specs/` 또는 관련 산출문서에 남긴다.

### 3. 구현 계획

- 구현은 작은 task로 나눈다.
- 각 task는 테스트, 구현, 검증, 커밋 단위로 끝나야 한다.
- 계획은 필요하면 `docs/superpowers/plans/`에 작성한다.

### 4. 코딩

- 기능 또는 버그 수정은 테스트를 먼저 작성한다.
- 기존 코드 스타일과 명명 규칙을 따른다.
- 실제 비밀번호와 secret은 `.env` 또는 서버 환경에만 둔다.
- `.env`, `.env.*` 파일은 커밋하지 않는다.

### 5. 리뷰

- 리뷰는 버그, 보안 위험, 요구사항 누락, 테스트 공백을 우선 확인한다.
- 의미 있는 작업은 `docs/reviews/YYYY-MM-DD-작업명-review.md`에 기록한다.
- PR 댓글에는 검증 결과와 남은 위험을 한글로 남긴다.

### 6. 테스트

기본 검증:

```bash
node tests\sheets.test.js
cd server
npm test
npm run build
npm audit --omit=dev
```

DB 변경이 있으면 개발 DB에서 migration과 seed를 확인한다.

```text
대상 DB: singziller_dev
대상 계정: devdbszusr
컨테이너: pdfowers-postgres-dev
```

### 7. 평가

- 완료 기준을 충족했는지 확인한다.
- 다음 세션으로 넘길 위험과 미완료 항목을 정리한다.
- 문서와 PR 상태가 실제 작업 상태와 일치하는지 확인한다.

### 8. 일정관리

- 완료된 항목은 `다음-작업-체크리스트.md`에서 완료 상태로 이동한다.
- 다음 우선순위는 한 세션에서 실행 가능한 단위로 정한다.
- 장기 과제는 `작업-일정.md`의 phase와 연결한다.

## 세션 시작 체크리스트

1. `git status --short --branch`로 브랜치와 미커밋 변경을 확인한다.
2. `git fetch --prune origin`으로 원격 상태를 갱신한다.
3. 현재 작업이 기존 PR의 연장인지, 새 Issue/PR이 필요한지 확인한다.
4. 관련 문서를 읽는다.
5. DB 또는 외부 시스템이 필요한 작업이면 접속 정보를 마스킹해 검증한다.

## 세션 종료 체크리스트

1. 테스트와 빌드를 실행한다.
2. DB 변경이 있으면 migration 적용 결과를 확인한다.
3. 리뷰 문서를 작성한다.
4. 문서와 체크리스트를 갱신한다.
5. 커밋하고 원격 브랜치에 push한다.
6. PR 또는 Issue에 결과 댓글을 남긴다.

## 에이전트 작업 프롬프트 기본형

```text
현재 브랜치와 작업트리를 확인하고, 관련 문서를 읽은 뒤 진행해.
작업은 기획, 설계, 구현, 리뷰, 테스트, 평가, 일정관리 흐름으로 정리해.
테스트를 먼저 작성하고, 의미 있는 단위로 커밋해.
실제 .env 값은 출력하지 말고 마스킹해서 확인해.
완료 후 docs/reviews와 다음 작업 체크리스트를 갱신하고 PR에 결과를 남겨.
```

## 현재 권장 운영 방식

- GitHub Issue를 실행 단위로 사용한다.
- PR은 `dev` 대상으로 만든다.
- 서버/DB 작업은 먼저 `dev` DB에서 검증한다.
- 공식 TJ/KY 수집은 robots.txt와 이용약관 확인 전까지 stub 또는 mock으로만 진행한다.
