# 브랜치 운영 프로세스

## 목적

Singziller 프로젝트의 세션 작업, PR, 승급, 기록 방식을 일관되게 운영한다. 모든 이슈, PR, 댓글, 리뷰 문서는 한글로 작성한다.

## 브랜치 구조

```text
dev  -> 개발 통합 브랜치
stg  -> 검증/스테이징 브랜치
main -> 운영/배포 기준 브랜치
```

## 기본 원칙

- Codex 로컬 세션 작업은 `dev` 브랜치 기준으로 진행한다.
- 기능 작업은 이슈를 먼저 등록한 뒤 시작한다.
- 기능 작업은 `dev`를 기준으로 작업 브랜치를 만들고 PR 대상도 `dev`로 둔다.
- 일반 세션 작업은 `#세션마무리` 흐름으로 종료한다.
- 브랜치 승급 작업은 `#브랜치승급` 흐름으로만 진행한다.
- `dev > stg`, `stg > main` 승급에는 추가 PR을 만들지 않는다.
- 승급은 fast-forward 또는 merge commit으로 처리하고, 이력은 댓글로 남긴다.
- 승급 댓글에는 포함된 PR 번호를 명시한다.
- 이슈, PR 제목, PR 본문, 댓글, 리뷰 문서는 모두 한글로 작성한다.

## 운영 키워드

### #세션마무리

일반 개발 세션을 마무리할 때 사용한다.

진행 순서:

1. 이슈를 먼저 등록한다.
2. `dev`에서 작업 브랜치를 생성한다.
3. 작업과 테스트를 완료한다.
4. 작업 리뷰 문서를 작성한다.
5. 작업 브랜치를 원격에 푸시한다.
6. `dev` 대상 PR을 생성한다.
7. PR 본문에는 관련 이슈, 변경 내용, 검증 결과, 남은 위험을 한글로 작성한다.

`#세션마무리`는 `dev`에 반영하기 위한 일반 작업 흐름이며, `stg` 또는 `main`으로 직접 반영하지 않는다.

### #브랜치승급

`dev > stg`, `stg > main` 승급 작업에만 사용한다.

진행 순서:

1. 승급 대상 범위와 포함 PR 번호를 확인한다.
2. 별도 PR을 만들지 않는다.
3. 대상 브랜치에 변경분을 반영한다.
4. 승급 이력을 댓글로 남긴다.
5. 댓글에는 포함 PR 번호, 승급 방향, 검증 결과를 반드시 작성한다.

`#브랜치승급`은 기능 개발이나 문서 수정 작업에 사용하지 않는다.

## 작업 시작 절차

1. 작업 내용을 GitHub Issue로 등록한다.
2. 이슈 본문에는 목적, 작업 범위, 완료 기준을 작성한다.
3. 작업 브랜치를 `dev`에서 생성한다.
4. 브랜치 이름은 `codex/이슈번호-작업명` 형식을 권장한다.
5. 작업 중 의사결정이나 범위 변경이 있으면 이슈 댓글에 남긴다.

## PR 절차

1. 작업 브랜치에서 변경을 완료한다.
2. 테스트와 검증을 실행한다.
3. `docs/reviews/`에 작업 리뷰 문서를 작성한다.
4. PR은 `dev` 브랜치를 대상으로 만든다.
5. PR 제목과 본문은 한글로 작성한다.
6. PR 본문에는 관련 이슈 번호, 변경 내용, 검증 결과, 남은 위험을 포함한다.
7. PR이 `dev`에 머지되면 이슈에 결과 댓글을 남긴다.

## 승급 절차

### dev에서 stg로 승급

1. `dev`에 포함된 승급 대상 PR 번호를 확인한다.
2. `stg` 브랜치에 `dev` 변경분을 반영한다.
3. 별도 PR은 만들지 않는다.
4. 관련 이슈 또는 승급 기록 이슈에 댓글을 남긴다.

댓글 예시:

```text
`dev`에서 `stg`로 승급했습니다.

포함 PR:
- #12 곡정보 다운로드 화면 추가
- #13 IndexedDB 카탈로그 저장소 추가

검증:
- 로컬 테스트 통과
- 스테이징 실행 확인
```

### stg에서 main으로 승급

1. `stg` 검증 완료 후 `main`에 반영한다.
2. 별도 PR은 만들지 않는다.
3. 승급 댓글에는 포함 PR 번호와 검증 결과를 남긴다.

댓글 예시:

```text
`stg`에서 `main`으로 승급했습니다.

포함 PR:
- #12 곡정보 다운로드 화면 추가
- #13 IndexedDB 카탈로그 저장소 추가

검증:
- 스테이징 검증 완료
- 배포 기준 문서 확인
```

## 커밋 메시지

커밋 메시지는 한글을 기본으로 한다.

형식:

```text
작업유형: 작업 내용 요약
```

예시:

```text
문서: 브랜치 운영 프로세스 추가
기능: 카탈로그 manifest 조회 추가
테스트: 시트 행 변환 테스트 보강
```

## 작업 리뷰 문서

모든 의미 있는 작업은 `docs/reviews/YYYY-MM-DD-작업명-review.md` 문서를 작성한다.

포함 항목:

- 작업 목적
- 변경 파일
- 의사결정
- 테스트/검증
- 남은 위험
- 다음 작업

## 세션 마무리 절차

세션 마무리는 `#세션마무리` 흐름으로 진행한다.

1. 변경 파일을 확인한다.
2. 테스트와 문법 검사를 실행한다.
3. 작업 리뷰 문서를 작성한다.
4. Git 상태를 확인한다.
5. 작업 브랜치에 커밋한다.
6. 작업 브랜치를 원격에 푸시한다.
7. `dev` 대상 PR을 생성한다.
8. 이슈/PR에 한글 댓글로 결과를 남긴다.

## 의사결정 필요

- 승급 이력 댓글을 개별 이슈에 남길지, 별도 `승급 기록` 이슈에 누적할지 결정해야 한다.
- `dev > stg`, `stg > main` 승급을 merge commit으로 할지 fast-forward로 할지 결정해야 한다.
- PR 머지 방식은 squash, merge commit, rebase 중 하나를 정해야 한다.

## 현재 권장값

- 승급 기록은 초기에는 관련 이슈 댓글에 남긴다.
- PR 머지는 `squash merge`를 권장한다.
- 승급은 브랜치 간 이력 추적을 위해 `merge commit`을 권장한다.
