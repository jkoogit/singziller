## 결정 필요

| 결정 | 필요 시점 | 현재 방향 |
| --- | --- | --- |
| IndexedDB 구현 방식 | Phase 2 구현 전 | 복잡도가 커지기 전까지 browser native IndexedDB 우선 |
| 카탈로그 payload 형식 | snapshot/delta publisher 전 | JSON snapshot과 JSON delta로 시작하고, 크기나 streaming 요구가 생기면 NDJSON 검토 |
| collector runtime과 배포 방식 | PostgreSQL 기반 collection run 전 | Node.js/TypeScript 서버 collector 유지 |
| TJ/KY live 접근 경계 | production source adapter 전 | probe와 compliance 검토 전에는 production scraping 금지 |
| `origin/dev`의 역할 | 향후 브랜치 운영 전 | 최신 workflow에 맞게 재정렬하거나 legacy integration branch로 정리 |

## 열린 리스크

| 리스크 | 영향 | 대응 |
| --- | --- | --- |
| 공식 TJ/KY 접근이 불안정하거나 제한될 수 있음 | live collection이 지연될 수 있다 | probe와 compliance 확인 전까지 adapter는 stub으로 유지 |
| `origin/dev`가 `stg`와 `main`보다 뒤처져 있음 | dev 대상 PR을 말하는 오래된 문서와 혼동될 수 있다 | AGENTS.md를 현재 기준으로 보고 오래된 문서를 점진 정리 |
| 일부 기존 문서에 인코딩 깨짐이 있음 | 문서 탐색과 상태 확인이 어렵다 | 새 운영 문서는 한글 파일명과 명확한 제목을 사용하고, 기존 문서 정리는 별도 작업으로 분리 |
| 수집 repository가 아직 in-memory임 | 실제 수집 작업으로 운영할 수 없다 | 다음 수집 slice에서 PostgreSQL repository 구현 |

## 운영 규칙

세션을 시작할 때 이 문서를 먼저 확인한다. PR이 `stg` 또는 `main`에 승급되거나 주요 작업 상태가 바뀌면 이 현황판을 함께 갱신한다.
