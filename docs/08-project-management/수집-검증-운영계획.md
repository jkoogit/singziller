# 수집 검증 운영계획

## 현재 결정

- 공식 API는 없는 것으로 보고 공식 홈페이지 검색 기능을 저속 샘플 수집한다.
- 개인 구현 API와 오픈소스 DB는 직접 의존하지 않고 후보/검증/설계 참고로 둔다.
- 초기 수집 주기는 혼합 정책으로 둔다.
  - 최신곡/차트: 주 1회
  - 전체 재검증: 월 1회
  - 개인 API 후보 재평가: 월 1회
- 최종 수집 주기는 2주 샘플 수집 지표로 결정한다.

## 역할별 의사결정 흐름

1. Collector Engineer가 source probe 결과를 남긴다.
2. Compliance Reviewer가 robots.txt, 약관, 요청량 위험을 확인한다.
3. Data Steward가 source tier와 trust score 기본값을 승인한다.
4. Validation Lead가 샘플 데이터의 정규화/중복/충돌 결과를 확인한다.
5. Operations Lead가 수집 주기와 pause/resume 여부를 결정한다.

## 샘플 수집 체크리스트

- TJ 반주곡 검색 parser fixture 확보.
- TJ 최신곡 parser fixture 확보.
- KY 공식 검색 경로 probe task 생성. 확정 전까지 KY production collector 구현 금지.
- 개인 API 후보의 license, repo URL, commit SHA, data schema 기록.
- source별 요청 간격과 user-agent 정책 정의.
- 403, 429, parser mismatch 발생 시 provider pause 처리.

## 검증 체크리스트

- 필수 필드: brand, title, artist, songNumber.
- 선택 필드: lyricist, composer, category, version, sourceUrl.
- 중복 기준: provider + payload hash, provider + external id.
- 충돌 기준:
  - 같은 brand/songNumber에 제목 또는 가수가 다름.
  - 같은 title/artist 후보가 TJ/KY에서 과도하게 많이 매칭됨.
  - 개인 API 결과가 공식 결과와 불일치함.
- 운영자 검수 큐 대상:
  - 필수 필드 누락
  - 낮은 trust score
  - 공식/개인 후보 충돌
  - 삭제 또는 변경 감지

## 진행 순서

1. 수집 provider 인터페이스와 mock provider 구현.
2. raw record hash와 중복 저장 방지 테스트 작성.
3. source probe 기록 구조 작성.
4. TJ 샘플 parser fixture 기반 단위 테스트 작성.
5. schedule policy 순수 함수 작성.
6. validation scoring 순수 함수 작성.
7. 검증 결과와 위험을 `docs/reviews/`에 기록.

## 완료 기준

- provider별 source policy가 문서화되어 있다.
- mock provider 수집 결과가 `source_records` 저장 흐름으로 검증된다.
- sample collection 결과는 자동으로 master catalog에 반영되지 않는다.
- schedule policy가 weekly/monthly/mixed를 결정하는 테스트를 가진다.
- validation scoring이 공식/개인/수동 충돌 케이스를 테스트한다.
