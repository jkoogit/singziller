# 작업 일정

## 전제

초기 목표는 PWA MVP를 유지하면서 PostgreSQL 기반 독자 카탈로그와 수집 모듈의 골격을 만드는 것이다. 일정은 2주 단위 phase로 나눈다.

## Phase 1: 문서와 서버 골격

기간: 1주

작업:

- 산출문서 정리.
- 서버 스택 결정: Node.js/TypeScript와 Fastify를 우선 적용.
- PostgreSQL migration 도구 결정: node-pg-migrate를 우선 적용.
- 환경별 DB 접속 설정 문서화.
- `server/` 프로젝트 골격 생성.
- `source_providers`, `collection_runs`, `source_records` migration 작성.
- `/catalog/manifest` mock API 작성.

완료 기준:

- 서버가 로컬에서 실행된다.
- PostgreSQL 연결 테스트가 통과한다.
- `dev`, `stg`, `main` 기준 DB 접속 설정 위치가 문서화된다.
- manifest mock 응답을 PWA에서 조회할 수 있다.

## Phase 2: IndexedDB 카탈로그 다운로드

기간: 1~2주

작업:

- PWA IndexedDB 래퍼 작성.
- `catalog_meta`, `catalog_songs`, `catalog_brand_entries`, `catalog_links` store 작성.
- 카탈로그 다운로드 화면 추가.
- manifest 조회, scope 선택, snapshot 저장 구현.
- 오프라인 검색을 IndexedDB 기준으로 전환.

완료 기준:

- 사용자가 scope를 선택해 다운로드할 수 있다.
- 오프라인에서 다운로드된 scope를 검색할 수 있다.
- 테스트가 통과한다.

## Phase 3: 수집 모듈 골격

기간: 1~2주

작업:

- `CollectorProvider` 인터페이스 작성.
- personal API/mock collector 작성.
- raw record 저장 구현.
- 중복 hash 계산 구현.
- collection run 로그 구현.
- normalizer 기본 규칙 작성.

완료 기준:

- mock source에서 곡정보를 수집해 `source_records`에 저장한다.
- 중복 raw record가 중복 저장되지 않는다.
- 수집 실행 이력이 기록된다.

## Phase 4: 카탈로그 생성과 검색 API

기간: 2주

작업:

- song master 생성 pipeline 작성.
- brand entry 생성.
- mapping candidate 생성.
- `/search` API 작성.
- snapshot/delta publisher 초안 작성.

완료 기준:

- 서버 검색 API가 정규화된 곡정보를 반환한다.
- PWA가 온라인 검색 결과를 표시한다.
- snapshot을 생성하고 다운로드할 수 있다.

## Phase 5: 공식/외부 소스 연결

기간: 별도 산정

작업:

- TJ 공식 검색 요청 방식 조사.
- KY 공식 검색 요청 방식 조사.
- YouTube Data API 보강 수집.
- 개인 수집 API adapter 연결.
- 검수 큐 운영 방식 확정.

완료 기준:

- 최소 1개 공식/외부 소스에서 실제 데이터를 수집한다.
- 수집 데이터가 raw/candidate/master 흐름을 거친다.

## 의사결정 필요

- Phase 2 전에 IndexedDB 라이브러리를 쓸지 순수 API로 구현할지 결정해야 한다.
- Phase 3 전에 collector runtime과 배포 방식을 결정해야 한다.

## 보완 필요

- 실제 개발 인력과 하루 작업 가능 시간.
- 운영 배포 자동화 방식.
- 공식 크롤링 법적 검토 일정.
