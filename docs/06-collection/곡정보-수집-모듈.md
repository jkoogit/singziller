# Collection Module Plan

## 목적

곡정보 수집 모듈은 앱과 분리된 서버 측 작업이다. 각 소스에서 원천 데이터를 수집하고, PostgreSQL에 raw record로 저장한 뒤 정규화와 매칭 작업을 거쳐 서비스 카탈로그에 반영할 후보를 만든다.

## 모듈 구성

```text
collectors/
  tj-official
  ky-official
  youtube
  personal-api
  google-sheet

pipeline/
  normalizer
  matcher
  scorer
  review-queue
  publisher
```

## 수집기 책임

### TJ 공식 수집기

- TJ 공식 검색/최신곡 페이지에서 곡번호, 제목, 가수, 작사가, 작곡가, 버전 정보를 수집한다.
- 요청 빈도 제한과 robots.txt를 준수한다.
- 공식 API가 확인되면 크롤링보다 API를 우선한다.

### KY 공식 수집기

- 금영/KY 공식 검색 접근 URL과 요청 방식을 확인한다.
- TJ와 동일한 정규화 스키마로 변환한다.
- 금영의 MR, 반주, 라이브 등 다중 버전을 별도 brand entry로 보존한다.

### YouTube 수집기

- 신규/미검수 곡에 대해서만 보강 검색한다.
- 공식 MV, 음원, 라이브, karaoke, MR 후보 링크를 수집한다.
- 곡번호와 브랜드 매핑의 확정 근거로 쓰지 않는다.

### 개인 수집 API

- 제공자별 신뢰도, 응답 포맷, 갱신 주기를 등록한다.
- API 응답 원본을 보관하고 정규화 결과와 분리한다.

### Google Sheet

- 운영자 또는 사용자가 검수한 곡정보를 import한다.
- 수동 확정 데이터로 볼지, 후보 데이터로 볼지 정책 결정이 필요하다.

## 정규화 규칙 초안

- 제목: 공백, 특수문자, 괄호 표기, OST 표기, remaster/live/MR 같은 버전 표기를 분리한다.
- 가수: feat., with, &, 콜라보 표기를 분리하고 alias 후보를 만든다.
- 번호: 브랜드별 원본 번호는 문자열로 저장한다.
- 버전: `original`, `accompaniment`, `mr`, `live`, `unknown` 중 하나로 정규화한다.
- 링크: YouTube/Spotify/기타 링크는 `song_links`로 분리한다.

## 현행화 주기 초안

- 최신곡 페이지: 1~3시간마다.
- 전체 공식 검색 재검증: 하루 1회.
- YouTube 보강: 신규/미검수 곡 큐 기반.
- 개인 API: 제공자별 SLA에 맞춤.
- 전체 매핑 재평가: 주 1회.

## 실패 처리

- 수집 실패는 `collection_runs`에 기록한다.
- 동일 소스가 연속 실패하면 해당 source를 일시 중지하고 알림 대상으로 올린다.
- raw record 저장은 성공했으나 정규화가 실패하면 review queue로 보낸다.

## 의사결정 필요

- 수집 모듈 런타임을 Node.js로 할지 Python으로 할지 결정해야 한다.
- 크롤링이 필요한 경우 Playwright 기반 동적 페이지 처리까지 포함할지 결정해야 한다.
- 관리자 검수 큐를 처음부터 UI로 만들지, DB 상태값으로만 운영할지 결정해야 한다.
- YouTube 검색어 템플릿을 어느 수준까지 자동화할지 결정해야 한다.

## 보완 필요

- TJ/KY 페이지별 HTML 구조와 요청 파라미터 조사.
- 각 소스의 이용약관/robots.txt 확인.
- 수집 요청 rate limit 기본값 정의.
- 중복 raw record 판정용 hash 규칙 정의.
- 가수 alias 사전과 제목 stopword 사전 설계.
